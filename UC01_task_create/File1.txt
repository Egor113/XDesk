				<p>В одной из записей (<a href="https://loadtestweb.wordpress.com/2017/10/17/%d1%83%d1%81%d1%82%d0%b0%d0%bd%d0%be%d0%b2%d0%ba%d0%b0-%d0%b8-%d0%bd%d0%b0%d1%81%d1%82%d1%80%d0%be%d0%b9%d0%ba%d0%b0-apache-jmeter/">Установка и настройка Apache.JMeter</a>) рассказывал про ручную установку и настройку Apache.JMeter. Сейчас расскажу про автоматизацию этого процесса.</p>
<p>В результате &#8212; все параметры настройки JMeter будут храниться в системе контроля версий. Тесты будут легко воспроизводимы, а процесс приблизится к тому, чтобы встроить нагрузку в CI/CD-конвейер.</p>
<pre>mvn verify или mvn jmeter:test</pre>
<p>И магия &#8212; скачался jmeter, плагины, запустился тест, сформировался отчёт.</p>

<h3>Назначение тегов в pom.xml</h3>
<p>Некоторые интересные теги в pom.xml выделил. На эти значения стоит обратить внимание или заменить на свои.</p>
<h4>Информация о проекте</h4>
<p>Обязательными атрибутами являются groupId и artifactId.</p>
<pre><span style="color:#333333;">    &lt;<span style="color:#003366;"><strong>groupId</strong></span>&gt;<span style="color:#800000;">com.loadtestweb</span>&lt;/<span style="color:#003366;"><strong>groupId</strong></span>&gt;
    &lt;<span style="color:#003366;"><strong>artifactId</strong></span>&gt;<span style="color:#800000;">jmeter-loadtestweb</span>&lt;/<span style="color:#003366;"><strong>artifactId</strong></span>&gt;</span></pre>
<p>Также нужно указать номер версии, название и описание теста.</p>
<pre><span style="color:#333333;">    &lt;<span style="color:#003366;"><strong>version</strong></span>&gt;<span style="color:#800000;">0.1.0</span>&lt;/<span style="color:#003366;"><strong>version</strong></span>&gt;</span>
<span style="color:#333333;">    &lt;<span style="color:#003366;"><strong>name</strong></span>&gt;<span style="color:#800000;">jmeter-loadtestweb</span>&lt;/<span style="color:#003366;"><strong>name</strong></span>&gt;</span>
<span style="color:#333333;">    &lt;<span style="color:#003366;"><strong>description</strong></span>&gt;<span style="color:#800000;">Apache.JMeter test for loadtestweb</span>&lt;/<span style="color:#003366;"><strong>description</strong></span>&gt;</span></pre>
<p>Выше указаны примеры значений. Их можно заменить на свои.</p>
<h4>Настройки</h4>
<p>Для удобства правки часто изменяющиеся параметры рекомендуется выносить в секцию <strong>properties</strong>. Так в эту секцию вынесено имя файла с тестом.</p>
<p>Сюда же можно вынести номер версии Apache.JMeter, версии плагинов &#8212; те параметры, которые с высокой вероятностью будут меняться в дальнейшем.</p>
<p>Позже в раздел <strong>properties</strong> мы добавим параметры профиля нагрузки &#8212; количество виртуальных пользователей, интенсивность выполнения.</p>
<pre><span style="color:#999999;">&lt;!-- Properties --&gt;</span>
<span style="color:#333333;">&lt;<span style="color:#003366;"><strong>properties</strong></span>&gt;</span>
<span style="color:#999999;">    &lt;!-- Test name --&gt;</span>
<span style="color:#333333;">    &lt;<span style="color:#003366;"><strong>performancetest.script</strong></span>&gt;<span style="color:#800000;">jmeter-loadtestweb.jmx</span>&lt;/<span style="color:#003366;"><strong>performancetest.script</strong></span>&gt;</span>
<span style="color:#333333;">&lt;/<span style="color:#003366;"><strong>properties</strong></span>&gt;</span></pre>
<p>Например, пусть наш jmx-файл будет называться jmeter-loadtestweb.jmx</p>
<pre>...
<span style="color:#333333;">&lt;<span style="color:#003366;"><strong>jMeterTestFile</strong></span>&gt;<span style="color:#008000;">${performancetest.script}</span>&lt;/<span style="color:#003366;"><strong>jMeterTestFile</strong></span>&gt;</span>
...
<span style="color:#333333;">&lt;<span style="color:#003366;"><strong>guiTestFile</strong></span>&gt;<span style="color:#008000;">${project.basedir}</span>/src/test/jmeter/<span style="color:#008000;">${performancetest.script}</span>&lt;/<span style="color:#003366;"><strong>guiTestFile</strong></span>&gt;</span>
...</pre>
<p>Это свойство можно использовать, чтобы указать путь к jmx-файлу для запуска теста в консольном и графическом режимах. Без дублирования информации.</p>
<p>Кроме того, любые параметры проекта, даже не переопределённые в pom.xml, можно использовать при формировании новых значений.</p>

<p>Теперь можно сохранить текущие изменения в git и отправить их во внешний репозиторий.</p>
<p>Предполагается, что первоначальная настройка git уже была выполнена.</p>
<p>Настроены имя пользователя и адрес электронной почты.</p>
<ul>
<li><a href="https://git-scm.com/book/ru/v2/%D0%92%D0%B2%D0%B5%D0%B4%D0%B5%D0%BD%D0%B8%D0%B5-%D0%9F%D0%B5%D1%80%D0%B2%D0%BE%D0%BD%D0%B0%D1%87%D0%B0%D0%BB%D1%8C%D0%BD%D0%B0%D1%8F-%D0%BD%D0%B0%D1%81%D1%82%D1%80%D0%BE%D0%B9%D0%BA%D0%B0-Git" target="_blank" rel="noopener">https://git-scm.com/book/ru/v2/Введение-Первоначальная-настройка-Git</a></li>
<li><a href="https://help.github.com/articles/setting-your-username-in-git/" target="_blank" rel="noopener">https://help.github.com/articles/setting-your-username-in-git/</a></li>
<li><a href="https://help.github.com/articles/setting-your-commit-email-address-in-git/" target="_blank" rel="noopener">https://help.github.com/articles/setting-your-commit-email-address-in-git/</a></li>
</ul>
<p>Удобно если имя и адрес электронной почты совпадают для всех проектов. Тогда можно действовать, как указано в инструкциях выше.</p>
<p>Текущие настройки можно посмотреть, выполнив команды</p>
<pre><span class="command" style="color:#333333;">git config user.name</span>
<span class="command" style="color:#333333;">git config user.email</span></pre>

<p>Если же для github.com, конкретно, используется специфичное имя и адрес электронной почты, то нужно указать специфичные настройки для текущего проекта.</p>
<p>Будет как в инструкциях выше, но без ключа &#8212;global.</p>
<pre class="command-line"><span class="command" style="color:#333333;">git config user.name "<span style="color:#800000;"><em>polarnik</em></span>"
git config user.email <span style="color:#800000;"><em>owasp@yandex.ru</em></span></span></pre>
<p>Теперь можно загитовать все изменения. И отправить их во внешнее хранилище.</p>

<h1>Настройка плагинов</h1>
<p>Допустим нам недостаточно базовых возможностей <strong>Apache.JMeter</strong> и решено использовать плагины.</p>
<p>Например плагин <strong>Ultimate Thread Group</strong>:</p>
<ul>
<li><a href="https://jmeter-plugins.org/?search=jpgc-casutg" target="_blank" rel="noopener">https://jmeter-plugins.org/?search=jpgc-casutg</a></li>
<li><a href="https://jmeter-plugins.org/wiki/UltimateThreadGroup/" target="_blank" rel="noopener">https://jmeter-plugins.org/wiki/UltimateThreadGroup/</a></li>
</ul>
<p>А конкретно версии 2.6.</p>
<p>В карточке Custom Thread Group сказано, что скачать плагины из этой группы можно используя артифакт</p>
<pre><span style="color:#333333;">Maven Artifact: kg.apc:jmeter-plugins-casutg:2.6</span></pre>
<p>Кроме того, скачав архив</p>
<ul>
<li><a href="https://jmeter-plugins.org/files/packages/jpgc-casutg-2.6.zip" target="_blank" rel="noopener">https://jmeter-plugins.org/files/packages/jpgc-casutg-2.6.zip</a></li>

<p>Таким образом, плагин jmeter-plugins-casutg-2.6.jar зависит от библиотеки jmeter-plugin-cmn-jmeter-0.4.jar, что важно для нас. И распространяется вместе с другим плагином jmeter-plugins-manager-1.3.jar, что не важно для нас.</p>
<p>В jar-файле с библиотекой написано как её найти в репозитории:</p>
<p>/META-INF/maven/kg.apc/jmeter-plugins-cmn-jmeter/pom.properties</p>
<pre><span style="color:#333333;">#Generated by Maven</span>
<span style="color:#333333;">#Tue May 02 15:16:47 MSK 2017</span>
<span style="color:#333333;">version=0.4</span>
<span style="color:#333333;">groupId=kg.apc</span>
<span style="color:#333333;">artifactId=jmeter-plugins-cmn-jmeter</span></pre>
<p>Поищем jmeter-plugin-cmn-jmeter-0.4.jar и найдём:</p>
<ul>
<li><a href="https://repo.maven.apache.org/maven2/kg/apc/jmeter-plugins-cmn-jmeter/" target="_blank" rel="noopener">https://repo.maven.apache.org/maven2/kg/apc/jmeter-plugins-cmn-jmeter/</a>
<ul>
<li>0.1</li>
<li>0.2</li>
<li>0.3</li>
<li>0.4</li>
<li>0.5</li>
<li>0.6</li>
</ul>
</li>
</ul>
<p>Значит kg.apc:jmeter-plugins-casutg:2.6 зависит от kg.apc:jmeter-plugins-cmn-jmeterg:0.4. Но в тоже время, в репозитории есть более новая версия библиотеки kg.apc:jmeter-plugins-cmn-jmeter:0.6.</p>
<pre><span style="color:#333333;">Maven Artifact: kg.apc:jmeter-plugins-cmn-jmeter:0.6</span></pre>
<p>Добавим в раздел конфигурации плагина jmeter-maven-plugin следующие секции:</p>
<pre><span style="color:#333333;">     &lt;<strong>jmeterExtensions</strong>&gt;</span>
<span style="color:#999999;">         &lt;!-- Custom Thread Groups</span>
<span style="color:#999999;">         https://jmeter-plugins.org/?search=jpgc-casutg</span>
<span style="color:#999999;">http://repo.maven.apache.org/maven2/kg/apc/jmeter-plugins-casutg/</span>
<span style="color:#999999;">--&gt;</span>
<span style="color:#333333;">         &lt;<strong>artifact</strong>&gt;kg.apc:jmeter-plugins-casutg:2.6&lt;/<strong>artifact</strong>&gt;</span>
<span style="color:#333333;">     &lt;/<strong>jmeterExtensions</strong>&gt;</span>

<span style="color:#333333;">     &lt;<strong>testPlanLibraries</strong>&gt;</span>
<span style="color:#999999;">         &lt;!-- JMeter Plugins Common Classes: Various utility classes to ease development of plugins</span>
<span style="color:#999999;">http://repo.maven.apache.org/maven2/kg/apc/jmeter-plugins-cmn-jmeter/</span>
<span style="color:#999999;">--&gt;</span>
<span style="color:#333333;">         &lt;<strong>artifact</strong>&gt;kg.apc:jmeter-plugins-cmn-jmeter:0.6&lt;/<strong>artifact</strong>&gt;</span>
<span style="color:#333333;">     &lt;/<strong>testPlanLibraries</strong>&gt;</span>

<span style="color:#333333;">     &lt;<strong>downloadExtensionDependencies</strong>&gt;false&lt;/<strong>downloadExtensionDependencies</strong>&gt;</span>
<span style="color:#333333;">     &lt;<strong>downloadLibraryDependencies</strong>&gt;false&lt;/<strong>downloadLibraryDependencies</strong>&gt;</span>
<span style="color:#333333;">     &lt;<strong>downloadOptionalDependencies</strong>&gt;false&lt;/<strong>downloadOptionalDependencies</strong>&gt;</span>
<span style="color:#333333;">     &lt;<strong>downloadJMeterDependencies</strong>&gt;false&lt;/<strong>downloadJMeterDependencies</strong>&gt;</span></pre>
<p>То есть:</p>
<ol>
<li>В секции <strong>jmeterExtensions</strong> добавлен плагин</li>
<li>В секции <strong>testPlanLibraries</strong> добавлены зависимости плагина</li>
<li>Указано, что все неявно заанные зависимости не нужно скачивать:
<ul>
<li><strong>downloadExtensionDependencies = false<br />
</strong></li>
<li><strong>downloadLibraryDependencies = false</strong></li>
<li><strong>downloadOptionalDependencies = false</strong></li>
<li><strong>downloadJMeterDependencies = false</strong></li>
</ul>
</li>
</ol>
<p>после чего скачаем плагины (mvn jmeter:confure).</p>
<p>Зафиксируем изменения в системе контроля версий.</p>
<p>Изменения в проекте отражены в правке:</p>
<ul>
<li><a href="https://github.com/polarnik/jmeter-loadtestweb/commit/ca53a2995ad8a1c2d3ea23b18ec28d6e7803afbb" target="_blank" rel="noopener">https://github.com/polarnik/jmeter-loadtestweb/commit/ca53a2995ad8a1c2d3ea23b18ec28d6e7803afbb</a> (Added jmeter-plugin UltimateThreadGroup)</li>
</ul>
<p>Таким образом можно добавить любые плагины в проект Apache.JMeter.</p>
<p>Их не придётся хранить вместе с проектом. Они всегда будут скачиваться при необходимости запуска. А в системе контроля версий будет храниться только указание скачать их в файле pom.xml.</p>
